# 系统架构

## 总体架构

```text
+------------------+
|  政务办公场景输入  |
+--------+---------+
         |
+--------v---------+     +-----------------+     +------------------+
|  Agent Skills    |---->|  ArbiterOS 内核  |---->|  Policy 判定结果  |
|  (Block 03)      |     |  (Block 05)     |     |  (Block 05)      |
+------------------+     +-----------------+     +------------------+
         |                                               |
         v                                               v
+------------------+                           +------------------+
|  红队 Case 库    |                           |  风险分级报告    |
|  (Block 01)      |                           |  (Block 04)      |
+------------------+                           +------------------+
         ^                                               |
         |                                               |
+------------------+                           +------------------+
|  攻击模式库      |---------------------------->|  系统展示/告警   |
|  (Block 02)      |                             |  (最终成果)    |
+------------------+                             +------------------+
```

## 模块说明

- **政务办公场景输入**：用户的自然语言指令或业务流程触发。
- **Agent Skills**：由 Block 03 设计，覆盖办文、办会、办事、邮件、文件等场景。
- **ArbiterOS 内核**： governance kernel，负责 policy 检查、trace 记录、风险拦截。
- **Policy 判定结果**：对每个 skill 调用进行 allow/block/ask 判定。
- **风险分级报告**：由 Block 04 输出，用于指导系统展示与响应策略。
- **攻击模式库**：由 Block 02 输出，持续更新并映射到 skills。
- **红队 Case 库**：由 Block 01 输出，用于持续测试与回归验证。

## 数据流

1. 用户输入触发某个 `gov_skill_*`。
2. Skill 调用请求送入 ArbiterOS 内核。
3. 内核根据 Block 02 攻击模式库和 Block 04 风险分级策略进行 policy 判定。
4. 判定结果记录到 trace，并返回给用户或拦截系统。
5. Block 05 定期批量运行 Block 01 的 case，验证系统防御能力。
